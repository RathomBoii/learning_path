★ Core Mindset

For every concept, always ask two questions: What problem does this solve? and What tradeoff does it introduce? — Data engineering is fundamentally about tradeoffs: speed vs cost, freshness vs reliability, simplicity vs scale.

01
Data Thinking Fundamentals
Before touching any tool — understand how data engineers think
›
Data vs Information vs Knowledge
Understanding what raw data actually is and how it transforms into actionable knowledge through processing and context.
Structured vs Semi-structured vs Unstructured
Why the shape of data fundamentally determines how it must be stored, processed, and queried.
Batch vs Streaming
The two fundamental processing paradigms — processing bounded data in chunks vs processing unbounded data continuously.
Data Producer vs Data Consumer
Understanding who creates data and who needs it — data engineers sit in between and serve both sides.
Data Freshness & Latency
How "old" can data be before it loses business value? The answer drives all pipeline design decisions.
02
Data Modeling
How to organize data so it's useful and queryable
›
Normalization vs Denormalization
Tradeoffs between write efficiency (normalized) and read efficiency (denormalized). A core tension in every modeling decision.
Relational Modeling
Entities, relationships, primary/foreign keys. The conceptual foundation even for non-relational systems.
Dimensional Modeling
Star schema, snowflake schema, fact tables, dimension tables. The dominant pattern for analytical data warehouses.
Slowly Changing Dimensions (SCD)
Types 1, 2, and 3 — strategies for tracking how dimension data changes over time without losing history.
Data Vault (Advanced)
Hub-satellite-link modeling for highly auditable, scalable enterprise data warehouses. Learn after mastering dimensional modeling.
03
Data Pipeline Concepts
How data moves from source to destination reliably
›
ETL vs ELT
Extract-Transform-Load vs Extract-Load-Transform. The order of operations changes who does the transformation work and why it matters at scale.
Idempotency
Running a pipeline twice should produce the same result. One of the most critical correctness properties for any data pipeline.
Incremental vs Full Load
Only processing new/changed data vs reprocessing everything. Incremental is efficient but complex; full loads are simple but expensive.
Backfilling
Reprocessing historical data when logic changes. A day-1 consideration when designing pipeline architecture.
Data Lineage
Tracking where data came from and how it was transformed — critical for debugging, compliance, and trust.
Late-Arriving Data
Handling data that shows up after it was expected. Common in event streams and IoT systems; requires watermarking and reprocessing strategies.
04
Storage Concepts
Where and how data lives — physically and logically
›
OLTP vs OLAP
Transactional (app) systems vs analytical systems. Fundamentally different design goals, read/write patterns, and scaling strategies.
Data Warehouse
Centralized, structured, schema-enforced storage optimized for analytical queries at scale.
Data Lake
Raw storage of any data type, any format, at any scale. Schema-on-read instead of schema-on-write.
Data Lakehouse
Hybrid architecture combining lake flexibility with warehouse performance. The dominant modern pattern.
Partitioning
Splitting data physically to enable query pruning — reads only the relevant partitions instead of full scans.
Columnar vs Row Storage
Why columnar formats (Parquet, ORC) are dramatically faster for analytical queries but slower for record-level writes.
Compression & Encoding
How data is physically stored efficiently — dictionary encoding, run-length encoding, and why they matter for both cost and speed.
05
Data Quality & Reliability
Making data trustworthy — treating pipelines as production systems
›
Data Quality Dimensions
Completeness, accuracy, consistency, timeliness, uniqueness — the five axes on which data quality is measured.
Schema Evolution
What happens when data structure changes over time — adding columns, changing types, removing fields — without breaking downstream consumers.
Data Contracts
Formal agreements between data producers and consumers on schema, SLAs, and expected data behavior.
Idempotency & Exactly-Once Semantics
Guaranteeing no duplicates and no data loss — the hardest problem in distributed pipeline design.
Monitoring & Alerting on Data
Treating pipelines like production software — detecting anomalies, schema drift, row count drops, and null rate spikes.
06
Distributed Computing Concepts
Processing data at scale — beyond a single machine
›
Why Distributed Computing?
Understanding single-machine limits — memory, CPU, and I/O ceilings — and why distributing work across nodes solves them.
Horizontal vs Vertical Scaling
Adding more machines vs upgrading existing ones. Data systems almost always scale horizontally by design.
Partitioning & Sharding
Splitting data across nodes by key, range, or hash. The choice of partition key is one of the most consequential design decisions.
MapReduce Paradigm
The foundational concept behind big data processing — split, process locally, then aggregate. Everything traces back to this model.
DAG — Directed Acyclic Graph
How pipelines and distributed computation are structured as dependency graphs — the backbone of every modern orchestration system.
Fault Tolerance & Retries
What happens when a node fails mid-pipeline — checkpointing, speculative execution, and idempotent task design.
07
Data Architecture Patterns
Putting it all together at system level
›
Lambda Architecture
Combining a batch layer (accurate, slow) and a speed layer (approximate, fast) to serve both historical and real-time queries.
Kappa Architecture
Stream-only simplification of Lambda — process everything as a stream, including historical reprocessing.
Medallion Architecture
Bronze (raw) → Silver (cleaned) → Gold (business-ready) data layers. The most adopted modern lakehouse pattern.
Event-Driven Architecture
How systems communicate through events rather than direct calls — the backbone of real-time data platforms.
Data Mesh
Decentralized, domain-owned data with federated governance. A cultural and organizational shift as much as a technical one.
